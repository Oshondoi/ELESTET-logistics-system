# System Patterns

## Architecture Style
Frontend-first MVP shell with a real Supabase-backed runtime.

Layers:
- Presentational React UI
- Feature-specific hooks / services
- Supabase client + RPC / table queries
- Supabase Auth + RLS-protected data model (dev bypass active)

## Main Frontend Structure
- `src/components/layout` — sidebar, topbar
- `src/components/shipments` — старые компоненты поставок (legacy, не удалять пока)
- `src/components/trips` — TripTable, TripFormModal
- `src/components/stores` — StoreFormModal, StoreList
- `src/components/accounts` — AccountFormModal, DeleteAccountModal
- `src/components/stickers` — StickerFormModal
- `src/components/ui` — Button, Badge, Card, Input, Modal, Select, Textarea
- `src/pages` — page composition only
- `src/hooks` — useAuth, useAccounts, useAppData
- `src/services` — shipmentService, storeService, tripService
- `src/lib` — supabase client, constants, utils, **stickerPdf**
- `src/types` — domain contracts

## State Pattern
1. resolve auth session (`useAuth`)
2. load user companies (`useAccounts`)
3. choose active company
4. load stores, shipments, trips from Supabase for active company (`useAppData`)
5. mutations through service helpers / RPC

## Domain Patterns

### Account / Company Model (КРИТИЧНО)
- **Account** = то, что пользователь видит как «Компания» в UI.
- Каждый пользователь может зарегистрироваться и создать свой account (компанию) независимо — это SaaS.
- `accounts.my_role` — роль текущего пользователя в этом account. Заполняется RPC `get_my_accounts`.
- После `createAccountWithOwnerInSupabase` объект возвращается **без `my_role`** — нужно вручную добавить `my_role: 'owner'` в `useAccounts.ts`.
- `useMyPermissions` полностью зависит от `my_role`: если undefined → `DEFAULT_PERMISSIONS` → пользователь видит только Главную.

### Multi-Tenant Boundary
`account_id` is the main tenant boundary for all business data.

### Sticker PDF Generation
```
stickerPdf.ts:
  renderStickerToCanvas(tpl) → canvas (580×400px, 10px/mm)
    HEADER (120px) — EAN-13 штрихкод (JsBarcode, width:4, flat:true) + цифры вручную
    BODY   (236px) — текст полная ширина: название 24px bold + поля 18px
    FOOTER  (44px) — 4 иконки по уходу 26px + ЕАС 26px, всё в ряд
  buildPdf(templates) → jsPDF (multi-page, copies учитываются)
  downloadStickerPdf / previewStickerPdf
```
- EAC рисуется через fillRect (не текст)
- ctx.save()/restore() во всех draw-функциях — изоляция состояния

### Trip → TripLine Hierarchy
```
Account
 └── Trip (Рейс #N — перевозчик, дата, статус)
      └── TripLine (Поставка M — магазин, склад, коробов)
```
- Trip number: unique within account (`unique(account_id, trip_number)`)
- TripLine shipment_number: unique within store (`unique(store_id, shipment_number)`)
- Generated by RPC: `create_trip`, `add_trip_line`

### Store-Specific Shipment Sequence (legacy)
Старые поставки (`shipments`) сохранены в БД. tracking_number уникален внутри store.
Новый UI работает с trips/trip_lines.

### Status History Pattern
Append-only history via `shipment_status_history` (для старых shipments).

### Safe Relational Design
Deletion is intentionally conservative; FK use restrict.

## UI Patterns
- Compact business layout
- Left sidebar: brand → company switcher → nav → logout
- Flat top bar with current page title
- Dense tables for operations
- Trip rows expand inline to show trip_lines
- Modal forms for creation flows
- Russian copy throughout the interface

## Important Boundaries
- Generation logic lives in DB / RPC; frontend previews only
- Minimal localized changes only; no opportunistic refactor
- Shared primitives changed only when task truly requires it
